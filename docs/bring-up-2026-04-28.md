# Bring-up — 2026-04-28

First-life validation of the dsp408-esphome external_component on an
ESP32-S3 DevKitC-1 talking to a directly-attached spare DSP-408 (no
USBIP, no Linux client in the path).

## Hardware

- **Bridge**: ESP32-S3, MAC `1c:db:d4:41:37:30`, rev v0.2, 240 MHz, 8 MB
  embedded PSRAM, on `/dev/cu.usbserial-230` (UART0 over the on-board
  USB-UART bridge).
- **DSP-408**: spare unit, serial `4EAA4B964C00` (the same unit that
  was being bridged via USBIP in earlier work), connected to the
  ESP32-S3's native USB-OTG port. Powered from the ESP32's USB host
  port (DSP-408 is bus-powered, ~50 mA).
- **No oscilloscope, no analyzer**, just serial logs over UART0.

## v0.1 enumeration & control surface

What worked on the first compile after writing the C++ from scratch:

```
[I][dsp408:051]: USB device opened — searching for HID interface...
[D][dsp408:123]: intf 0: class=0x03 subclass=0x00 proto=0x00 eps=2
[D][dsp408:137]:   ep 0: addr=0x82 attr=0x03 mps=64
[D][dsp408:137]:   ep 1: addr=0x01 attr=0x03 mps=64
[I][dsp408:070]: Claimed HID interface 0, EP-IN=0x82 EP-OUT=0x01
[V][dsp408:315]: -> dir=0xA2 cat=0x09 cmd=0xCC seq=1 len=0
[V][dsp408:272]: <- dir=0x53 cat=0x09 cmd=0xCC seq=1 len=1
[I][dsp408:386]: CONNECT ok (status=0x00)
[V][dsp408:315]: -> dir=0xA2 cat=0x09 cmd=0x4 seq=2 len=0
[V][dsp408:272]: <- dir=0x53 cat=0x09 cmd=0x4 seq=2 len=11
[I][dsp408:401]: GET_INFO: 'MYDW-AV1.06'
[V][dsp408:315]: -> dir=0xA2 cat=0x09 cmd=0x5 seq=3 len=0
[V][dsp408:272]: <- dir=0x53 cat=0x09 cmd=0x5 seq=3 len=8
[I][dsp408:421]: MASTER read: +0 dB (audible)
[I][dsp408:433]: DSP-408 ready (startup took 101 ms)
```

That's the entire wire-level handshake exactly mirroring dsp408-py:
- CONNECT (cmd=0xCC, cat=0x09) → 1-byte status reply (0x00 = ok)
- GET_INFO (cmd=0x04, cat=0x09) → 11-byte ASCII identity string
- MASTER read (cmd=0x05, cat=0x09) → 8-byte payload with level + mute

**Total time from `setup() finished successfully!` to `DSP-408 ready`:
624 ms.** The first 310 ms is the USB host stack enumerating the
device (`usb_host took a long time for an operation (310 ms)`); the
DSP-408 protocol handshake itself is 101 ms.

## v0.1 write path

A self-driving on_boot script exercised every write path:

| Action                              | Submit time | Ack time | Round-trip |
| ----------------------------------- | ----------- | -------- | ---------- |
| `set_master(-20 dB, audible)`       | 0.000 s     | 0.038 s  | 38 ms      |
| `set_master(-20 dB, muted)`         | 0.000 s     | 0.036 s  | 36 ms      |
| `set_master(0 dB, audible)`         | 0.000 s     | 0.036 s  | 36 ms      |
| `set_master(-5 dB, audible)`        | 0.000 s     | 0.036 s  | 36 ms      |
| `set_channel(0, -15 dB, audible)`   | 0.000 s     | 0.005 s  | 5 ms       |
| `set_channel(0, -15 dB, muted)`     | 0.000 s     | 0.011 s  | 11 ms      |
| `set_channel(0, -15 dB, audible)`   | 0.000 s     | 0.006 s  | 6 ms       |
| `set_channel(0, 0 dB, audible)`     | 0.000 s     | 0.015 s  | 15 ms      |
| `set_master(0 dB, audible)` final   | 0.000 s     | 0.036 s  | 36 ms      |

Every request was acknowledged. The slower ~36 ms master writes vs the
~5–15 ms channel writes lines up with the dsp408-py observation that
master takes longer because the firmware actually triggers a state
re-broadcast for it.

Periodic 5-second master polls confirmed the device-side state matched
each requested value: -20 dB → muted → -5 dB → +0 dB end of test, with
the periodic re-read entity update reflecting it back to HA.

## What's not in v0.1

(All slated for follow-on; tracked in [README.md](../README.md).)

- 296-byte multi-frame channel state read (`cmd=0x77NN`) — needed for
  state recovery on reconnect and for reading current EQ/crossover.
- Per-channel delay (the wire format already encodes it; just needs
  exposure as a `number:` entity).
- Crossover (HPF + LPF, `cmd=0x12000+ch`).
- 10-band parametric EQ (`cmd=0x10000 + (band<<8) + ch`).
- Output routing matrix (`cmd=0x2100+ch`).
- Preset name read/write.
- Compressor / dynamics (`cmd=0x2300+ch`).
- Input-side processing (the cat=0x03 plane).

## Notes for follow-on work

- **Channel writes are fire-and-forget** in v0.1 — we only get a 0-byte
  ack, no read-back. dsp408-py works around this with an in-memory
  cache of the desired per-channel state; we inherited the same
  pattern. State is RAM-only and rebuilt on each connect; users
  setting a channel volume from HA before the channel state has been
  primed get default subidx values. Persisting to flash via ESPHome's
  preferences API would close that gap.
- **Startup write-drop quirk** (per dsp408-py): the firmware silently
  drops the first ~5–6 cmd=0x1FNN writes after a fresh session if
  they arrive too fast. v0.1 just doesn't issue any per-channel writes
  on connect — they only fire on user input. If we ever add a "warmup
  reads" step (8 channel reads to drain the queue) it should land in
  the startup state machine before transitioning to RUNNING.
- **No timeout retry** in v0.1 for in-flight requests — if a reply is
  dropped, the watchdog kicks the whole startup back to CMD_CONNECT
  rather than retrying just the lost command. Good enough for v0.1
  because it's already a hard problem path; should be replaced with
  per-cmd retry once we add multi-frame reads.

## Compared to USBIP path

| Metric                          | USBIP-via-WiFi | Direct USB host |
| ------------------------------- | -------------- | --------------- |
| Boot-to-first-control           | ~10–15 s       | **624 ms**      |
| Master-write round-trip         | ~80–150 ms     | **36 ms**       |
| Channel-write round-trip        | ~60–120 ms     | **5–15 ms**     |
| Failure modes outside our reach | kernel + TCP   | none            |
| Survival under burst load       | poor           | clean (so far)  |
| Code complexity (LOC, fw side)  | ~3000 (full bridge) | ~600 |

The latency difference is dominated by removing the
`HA → MQTT → dsp408-mqtt → hidapi → kernel HID → vhci-hcd → TCP/IP →
ESP32 firmware → usb-host` chain (10 hops) in favor of just
`HA → ESPHome API → entity → DSP408 component → usb_host → DSP-408`
(5 hops, half of which are in-process).
